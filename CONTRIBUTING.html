    Contributing Notes

    There are some things to consider when contributing to this repo:

    • make screenshots in high resolution by temporarily increasing the font scaling factor in your OS and using a large display

    • limit the width of images and center them, for example, as follows:

      ![](images/my_image.png){width=500px fig-align="center"}

    • annotate code blocks with filename="Terminal" andfilename="Console" to differentiate Bash and R that should be entered by the user (if there is no file from which the code is an excerpt)

    • Put optional content into a collapsed note

    • Put important content into warning

    • hierarchy of callouts

      • note
      • tip
      • caution
      • warning
      • important

    • headings (including tabset tabs and callout titles) are written in Title Case

    • on any page, start headings at level 2 (i.e., use ##) and above because the page title is already at level 1

    • maintain the hierarchy of headings, that is, only create headings at level n if thereâ€™s already a heading at level n-1

    • also maintain the hierarchy of headings for callouts: callouts use heading levels that are one below the current heading level

    • R package names are typeset in monospace

    • Prefer R packages that follow the tinyverse philosophy, that is, have a lightweight dependency footprint. All packages need to be downloaded by renv and (worst case) compiled during the tutorial, which better not take too long. Use pak::pkg_deps() to count the total number of package dependencies.

    • Write license abbreviations with no-break spaces (e.g., write CC\ BY\ 4.0 rather than CC BY 4.0)

    • When writing markdown:

      • use semantic linefeeds to ease change tracking, except for exact quotes (as they wonâ€™t be updated) and for example code in code blocks (to allow for wrapping on the website as required by the userâ€™s device)
      • use regular footnotes rather than inline footnotes to increase the readability of markdown (except for short notes or at the end of lines)
      • use reference links rather than inline links to increase the readability of markdown (except for short links or at the end of a line)

    • If the (R package/system) dependencies that are required to work through the tutorial change, make sure to adjust the respective section in make_readme.qmd accordingly.

    • When making significant contributions, add people to the CITATION.cff file and mention them in about.qmd.

    • keep usage of terms consistent, for example:

      • always write data dictionary or always write codebook (but donâ€™t mix)
        • in the case of this tutorial, the convention is to write data dictionary
      • always write data set or always write dataset (but donâ€™t mix)
        • in the case of this tutorial, the convention is to write data set
